  5

  9  Known Limitations 
 10  ================= 
 11   
 12  array index assignment 
 13  ----------------------   
 14   
 15  in mel, you can directly assign the value of any element in an array, and all intermediate elements will be  
 16  automatically filled. this is not the case in python: if the list index is out of range you will raise an  
 17  IndexError. i've added fixes for several common array assignments:: 
 18   
 19  append new element 
 20  ~~~~~~~~~~~~~~~~~~ 
 21   
 22  MEL 
 23          >>> $strArray[`size $strArray`] = "foo"; 
 24           
 25  Python   
 26          >>> strArray.append("foo") 
 27   
 28  assignment relative to end of array 
 29  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ 
 30   
 31  MEL 
 32          >>> $strArray[`size $strArray`-3] = "foo"; 
 33           
 34  Python   
 35          >>> strArray[-3] = "foo" 
 36   
 37  However, since the translator does not track values of variables, it does not know if any given index is out of 
 38  range or not. so, the following would raise an 'list assignment index out of range' error when converted to 
 39  python and would need to be manually fixed:: 
 40   
 41          string $strArray[]; 
 42          for ($i=0; $i<5; $i++) 
 43                  $strArray[$i] = "foo" 
 44           
 45                                           
 46  for(init; condition; update) 
 47  ---------------------------- 
 48   
 49          the closest equivalent to this in python is something akin to:: 
 50   
 51                  for i in range(start, end) 
 52                   
 53          in order for this type of for loop to be translated it must meet several requirements: 
 54           
 55                  1. the inititalization, conidition, and update expressions must not be empty. 
 56                           
 57                          not translatable:: 
 58                                  for(; ; $i++) 
 59                   
 60                  2. there can be only one conditional expression.    
 61                           
 62                          not translatable:: 
 63                                  for($i=0; $i<10, $j<20; $i++) 
 64                   
 65                  3. the variable which is being updated and tested in the condition (aka, the iterator) must exist alone on one 
 66                          side of the     conditional expression. this one is easy enough to fix, just do some algebra: 
 67                           
 68                          not translatable:: 
 69                                  for($i=0; ($i-2)<10, $i++) 
 70           
 71                          translatable:: 
 72                                  for($i=0; $i<(10+2), $i++) 
 73                   
 74                  4. the iterator can appear only once in the update expression: 
 75                           
 76                          not translatable:: 
 77                                  for($i=0; $i<10, $i++, $i+=2) 
 78   
 79  Inconveniences 
 80  ============== 
 81   
 82  switch statements 
 83  -----------------                
 84  alas, switch statements are not supported by python. the translator will convert them into an if/elif/else statement. 
 85   
 86   
 87  global variables 
 88  ----------------                 
 89   
 90  Global variables are not shared between mel and python.  two functions have been added to pymel for this purpose: 
 91  L{getMelGlobal} and L{setMelGlobal}. by default, the translator will convert mel global variables into python global  
 92  variables AND intialize them to the value of their corresponding mel global variable using getMelGlobal(). if your  
 93  python global variable does not need to be shared with other mel scripts, you can remove the get- and 
 94  setMelGlobals lines (for how to filter global variables, see below). however, if it does need to be shared, it is very 
 95  important that you manually add setMelGlobal() to update the variable in the mel environment before calling any mel  
 96  procedures that will use the global variable.  
 97   
 98  In order to hone the accuracy of the translation of global variables, you will find two dictionary parameters below -- 
 99  L{global_var_include_regex} and L{global_var_exclude_regex} -- which you can use to set a regular expression string 
100  to tell the translator which global variables to share with the mel environment (i.e. which will use the get and set 
101  methods described above) and which to not.  for instance, in my case, it is desirable for all of maya's global  
102  variables to be initialized from their mel value but for our in-house variables not to be, since the latter are often 
103  used to pass values within a single script. see below for the actual regular expressions used to accomplish this. 
104                                                   
105                                                   
106  Command Callbacks 
107  ----------------- 
108   
109  one common point of confusion is command callbacks with ui elements. There are several different ways to handle  
110  command callbacks on user interface widgets:   
111                                                   
112  Function Name as String 
113  ~~~~~~~~~~~~~~~~~~~~~~~ 
114   
115  maya will try to execute this as a python command, but unless you know the namespace it will 
116  be imported into, the function will not be recognized. notice how the namespace must be hardwired 
117  into the command: 
118   
119          >>> button( c="myCommand" ) 
120   
121  or 
122   
123          >>> button( c="myModule.myCommand" ) 
124   
125  this method is not recommended. 
126   
127  Function Object 
128  ~~~~~~~~~~~~~~~   
129   
130  When using this method, you pass an actual function object (without the parentheses). The callback function 
131  has to be defined before it is referenced by the command flag.  also, keep in mind many ui widgets such as radioButtonGrp 
132  pass args to the function (to mimic the "myCommand #1" functionality in mel), which your function must accommodate. The 
133  tricky part is that different ui elements pass differing numbers of args to their callbacks, and some pass none at all. 
134  This is why it is best for your command to use the *args syntax to accept any quantity of args, and then deal with them 
135  in the function. 
136   
137          >>> def myCommand( *args ): print args # this definition must come first 
138   
139          >>> button( c=myCommand ) 
140                                   
141  Lambda Functions 
142  ~~~~~~~~~~~~~~~~ 
143  In my experience this is the best way to handle most command callbacks.  You can choose exactly which args you want 
144  to pass along to your function and order of definition does not matter. 
145   
146          >>> button( c= lambda *args: myCommand(args[0]) ) 
147   
148          >>> def myCommand( arg ): print "running", arg  
149   
150                                                   
151                                                   
152  comments 
153  -------- 
154  rules on where comments may be placed is more strict in python, so expect your comments to be shifted around a bit 
155  after translation. 
156   
157   
158  formatting 
159  ---------- 
160                                   
161  the formatting of your original script will be lost. i apologize for this, and perhaps i'll be able to carry over some 
162  custom formatting in future versions, but python is much more strict about formatting than mel, so the conversion is 
163  infinitely simpler when the formatting is completely discarded and reconstructed based on pythonic rules. 
164

166  Solutions and Caveats 
167  ===================== 
168   
169  catch and catchQuiet     
170  -------------------- 
171   
172  there is no direct equivalent in python to the catch command and it does not exist in maya.cmds so i wrote two 
173  python commands of the same name and put them into pymel. these are provided primarily for compatibility with  
174  automatically translated scripts. try/except statements should be used instead of catch or catchQuiet if coding 
175  from scratch.                    
176   
177  for( $elem in $list ) 
178  --------------------- 
179           
180  this variety of for loop has a direct syntactical equivalent in python.  the only catch here is that maya.cmds  
181  functions which are supposed to return lists, return None when there are no matches. life would be much simpler  
182  if they returned empty lists instead.  the solution currently lies in pymel, where i have begun 
183  correcting all of these command to return proper results.  i've started with the obvious ones, but there  
184  are many more that i need to fix.  you'll know you hit the problem when you get this error: 'TypeError: iteration 
185  over non-sequence'. just email me with commands that are giving you problems and i'll fix them as 
186  quickly as i can.
